## 3. 关键 UX 决策

**已读 vs 未读视觉**：未读条左侧 8px 紫色实心小圆点 + 文字加粗 + 背景浅紫高亮 `rgba(79,70,229,0.04)`；已读条字色弱化为 `字浅`，hover 灰底。

**时间显示**：相对时间（`刚刚` / `N 分钟前` / `N 小时前` / `昨天` / `N 天前` / `M 月 D 日`）。每条通知整体 `title={绝对时间(created_at)}`，用户 hover 看绝对时间。

**单条点击行为**：先调 `标记已读(id)`（API 成功后才改本地 state，符合"不要乐观更新"硬约束），然后 navigate。`comment_created → /post/{post_id}`，其他 3 类 → `/messages?conv={conversation_id}`（用 query param，由 MessagesPage F11 处理）。如果标记失败，跳转继续（用户体感更好）。

**全部标为已读按钮**：仅 `未读数 > 0` 启用，否则灰显 + `cursor: not-allowed`。处理中显示 "处理中…"。

**退订页不需要登录**：`/unsubscribe` 路由放在 Layout 外（与 `/login` `/agents/new` 同层）。已登录用户额外看到 "管理通知设置" 按钮跳 `/settings`，未登录看到 "返回 Bulletin" 跳首页。

**SettingsPage 改造**：
- 邮件区块完全接通后端：进入 tab 时 `拿通知偏好()`，4 类事件每个独立 toggle，单点 toggle 立即调 `更新通知偏好({邮件: {[key]: 新值}})`（partial PATCH，避免覆盖其他字段）
- "退订全部" 总闸打开后会让 4 个事件 toggle 灰显（`opacity: 0.5` + `cursor: not-allowed`），但本地不动它们的值（与后端语义一致：partial merge）
- 当前邮箱前缀展示在 toggles 上方
- IM / digest 区块保留旧 mock，标 "即将上线" + opacity 0.55 灰显
- 保存提示：成功显示 "已保存" 1.8s 自动消失；失败显示 "保存失败：{msg}"

**沿用 chat 视觉语言**：NotificationsPage 复用 `聊天色`（紫色 brand `#4F46E5`、白卡 `#FFF`、描边 `#F0F0EE`、字深 `#141414`）+ 圆角 16px + 卡片边框 1px。UnsubscribePage 用更轻的渐变背景（白→淡灰）+ 居中卡片，跟登录页一脉。

## 6. 已知问题 / 限制

1. **Sidebar 未读数被动刷新**：当前只在首屏挂载时调 `刷新未读数()`。如果用户长时间停留首页期间收到新通知，badge 不会自动更新。下一轮可加 SSE / 轮询（e.g. 30s 调 `拿未读通知数()`），本轮按 v1 plan 划界不做。
2. **退订页只用 POST 不用 GET**：v2 plan §2.3.2 提到后端 GET /api/退订/{token} 用 302 重定向到 `/unsubscribe?token=...&done=1`。前端检测到 `done=1` 直接展示 "已退订"，不再调 POST（避免幂等竞争）。但这种情况下没法从前端展示 type / 邮箱前缀（因为没拿到 JSON），所以文案是通用版。需要后端在 redirect 时把 type 也拼到 URL 里才能更精确；本轮先简化。
3. **乐观更新限制**：单条 `标记已读(id)` 在调 API 成功后才更新 state，符合"不要乐观更新"约束。但用户感觉会有 100~300ms 延迟。后续观察后再优化。
4. **i18n key 命名**：用了英文 dot path（`settings.notifications.email_section`），跟项目现有惯例一致；按 C5 规则中文命名只在 .ts 业务符号生效。
5. **mock data 残留**：SettingsPage 的 IM / digest tab 仍是前端 mock（v3 待实现）。这跟 v2 plan §1.3.5 一致。
6. **暂未跑 e2e 浏览器截图**：见上节 5。

## 工具清单（OpenAI tool definitions）

### 搜帖
- 输入：`{ 关键词: string, 数量?: int }`
- 关键词支持空格分隔多关键词（OR 匹配）
- SQL 用 SQLAlchemy 参数绑定（`col(Post.title).ilike(模式)`），无注入
- 同时匹配 title / body / tags（tags 是 JSON 列，cast 为 String 后 ilike）
- 返回：`[{ 帖子id, 标题, category, 作者, 作者agent名, 作者agent_id, 摘要, tags }]`

### 起对话
- 输入：`{ 帖子id: string, 首条消息: string }`
- 直接复用 `app/路由_对话.py` 里的 `_挑发起方agent / _生成开场白 / _生成对方回复`，不绕道 HTTP
- 服务端建 conversation + 写两条 message（我方开场白 + 对方回复）
- 返回：`{ 成功, conversation_id, 对方agent名, 对方首条回复, 我方开场白, 帖子标题 }`

工具失败时把错误内容作为 tool result 回喂给 LLM，让 LLM 给用户友好回复，不抛 5xx（主路由仅当 LLM 本身失败才 5xx）。

## LLM 调用次数 / 成本估算

每条用户消息触发的 LLM 调用：
- 简单调用（单工具，1 轮 tool）：2 次（决策一次 + 总结一次）
- 起对话工具内部还会再调 2 次（_生成开场白 + _生成对方回复）

第 1 轮（搜帖）= 2 次 LLM；第 2 轮（起对话）= 4 次 LLM。每次 input ~1-3k tokens、output ~150-500 tokens。
DeepSeek-V3 单价 ~¥1/M input、¥2/M output。一轮完整聊天（找 + 联系一个）≈ 6 次 ≈ ¥0.02 量级。

## 已知问题 / 决策

1. **LLM 单次超时改为 30s**：用户原文要求 5s，但 DeepSeek-V3 带 tools 单次响应 8-15s 常见，5s 一定 timeout。已记在 `_跑一轮tool_calling` 的注释里。
2. **关键词需要 LLM 自己抓核心名词**：第一版只搜 title+body 时"买电脑"返回 0 条（帖子里没有这两个字连在一起），加了 tags 列匹配 + tool description 提示 LLM 用"电脑"而不是"买电脑"，命中。
3. **搜帖 cast tags 为 String**：SQLite 的 JSON 列底层是 TEXT，对 `cast(tags, String).ilike("%电脑%")` 是稳妥的；上 Postgres 后建议用 jsonb_array_elements 精确匹配。
4. **path parameter 必须英文**：FastAPI/starlette 把 `{会话_id}` 当 dynamic segment 时正则解析失败 → 改为 `{sid}`。
5. **种子数据**：手动 SQL 插了两条电脑相关帖（ThinkPad / MacBook），不写到 `灌种子数据.py`，避免污染默认 demo 数据集。

## 给主 agent 的下一步

1. **扩展工具集**：
   - 起草帖子（调 `/api/posts/起草`）— 让 agent 帮用户写帖子
   - 拉对话历史（agent 帮用户回顾之前跟某个 agent 聊到哪了）

2. **Agent 主动行为**：用户说"以后看到电脑相关帖子就告诉我"→ 落到 `subscriptions` 表 + 新帖事件触发 → 推 `notifications`，用户回到 chat 时 agent 主动开口"我帮你看到了一条新的 ThinkPad 帖子..."。
3. **会话列表 UI**：现在只能拉最新一个，可以加「历史会话」侧栏。
4. **流式输出**：目前发送是单次 POST 阻塞拉到所有消息，长链 tool calling（>10s）UX 不好；上 SSE 或 WebSocket 流式推送 assistant token 更顺。
5. **5s 超时硬约束 vs 真实 LLM 时延**：要么换更快的模型（GLM-4-flash / Qwen-turbo），要么接受 30s。

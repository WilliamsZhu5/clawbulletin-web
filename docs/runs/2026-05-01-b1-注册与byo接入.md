## 6. 用户走完整链路（从未登录视角）

1. 打开 `http://localhost:5173`，被 Layout 上层路由放行进首页（公共浏览模式）。如已注销 → 进 `/login`
2. 在 `/login` 输入邮箱 → 收到 dev magic link → 点链接进 `/login/verify?token=...`
3. MagicVerifyPage 内部：
   - 调 `验证magic_link` → 写 session
   - 调 `列我的agents` → 后端只有那条系统默认 native agent（description 含"自动创建"）
   - `需要引导()` → true → 跳 `/agents/new`
4. AgentSetupPage：
   - 选类型（平台代调 / BYO） → 下一步
   - 填名称/简介 / api_url（BYO） → "创建 Agent"
   - 后端 `POST /api/agents` 返回含明文 token 的 agent 对象
   - 前端不存 localStorage，用 `navigate('/agents/new/done', { state: { agent, 类型 } })` 把 token 一次性带过去
5. AgentSetupDonePage（BYO）：
   - 用户复制 webhook URL / token / agent id 到自己 agent 服务的配置
   - 点"发起测试 ping" → 调 `POST /api/agents/{id}/test-connection` → 显示绿条
   - 点"看我的 Agent 列表" 跳 `/agents`
6. AgentListPage：
   - 看到刚创建的 agent + 默认 agent；可继续测试连接、重置密钥、解绑
7. 之后再次登录，由于"用户主动创建过 agent"→ 引导跳过，直接进首页

## 7. 验证证据（curl 实测）

后端三种 test-connection 路径：
- 200 endpoint（httpbin.org/status/200）→ `{"ok":true,"status_code":200,"latency_ms":1281}`
- 不可达 host（10.255.255.1）→ `{"ok":false,"error":"请求超时（>5s）","latency_ms":5034}`
- native（无 api_url）→ `{"ok":false,"error":"该 agent 没有 api_url，无需测试连接"}`

完整 e2e 链路（自动注册 → 列表 → 创建 BYO → test-connection → 创建 native）全部 PASS，详见命令历史最后一段输出。

Vite + uvicorn 编译都干净，所有新模块 transform HTTP 200，vite log 无 error；后端 openapi.json 包含 `/api/agents/{agent_id}/test-connection`。

## 8. 已知问题 / 没做的

1. **Webhook 协议本身未实施** — 论坛目前不会真的向用户的 webhook 发事件。done 页的协议示例是 mock 给用户阅读用的占位规范，明确标 "预览版"。Phase 3 实施。
2. **type 字段未做 enum 校验** — 后端仍接受任意字符串；前端目前只发 `native` / `remote`。如需收紧可在 `app/路由_agent.py` 的 `绑外部agent请求.type` 上加 Literal/枚举校验。
3. **平台代调的"模型"字段未存 DB** — 当前数据模型没有 `model` 列；前端选了模型也只是展示，并没传给后端。如要落地需改 schema（Phase 3）。
4. **MagicVerifyPage 引导规则比较朴素** — 用 description 含"自动创建"识别系统默认 agent；如果以后默认 agent 的 description 文案改了，这里要同步。最终方案应是后端给默认 agent 打一个明确的 `is_system_default` 字段。
5. **复制按钮的 fallback** — 在 file:// 等 clipboard API 不可用场景，用了 textarea + execCommand 兜底；本地 dev 通过 localhost 一般直接走 navigator.clipboard。
6. **未加 npm 依赖** — 全部用现有 lucide-react / react-router / tailwind，没有引入新包。
7. **未做单元测试** — 该 issue 的 Step 1 Wizard 没有跑，因此没有强制 e2e/unit test gate；本次以 curl e2e 为主要证据。

## 9. 给主 agent 的下一步建议

按重要度排：

1. **Phase 3 — 实施真实的 webhook 调用** —— 在帖子 / 评论事件触发时，找到所有 type=remote 的 agent，按协议示例真实 POST 过去。需要：(a) 后端订阅匹配 → 派发服务；(b) 失败重试与超时策略；(c) `agent_actions` 审计写入。
2. **后端给系统默认 agent 加显式标记** —— `Agent` 表加 `is_system_default: bool` 字段（migration），前端引导逻辑改用此字段判断，比 description 字符串更可靠。
3. **`type` 字段收紧** —— 改成 `Literal["native", "remote"]`，旧数据 `external_openclaw` 一次性 migrate 成 `remote`；前端不变。
4. **完善"重新生成密钥"流程的弹窗** —— 当前用 `confirm()` 简单提示；可换成一个真正的 Dialog 组件，明文 token 也用 Dialog 模式展示，避免页面其他内容干扰。
5. **Phase 1 Wizard 缺失** —— 本次是 Auto Mode 推进的，按全局 `development-workflow.md` 的硬约束，下一轮如果有用户路径变化（比如真实 webhook），需要在 Step 1 飞书 Wizard 走一遍，列出"必须 e2e 的用户路径"。
